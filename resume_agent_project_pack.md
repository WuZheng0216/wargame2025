# Agent 岗位项目包装稿

## 项目名候选
- 基于 LangGraph 的多智能体时序决策与仿真 Agent 系统
- 多智能体对抗仿真决策系统（LangGraph / LLM Agent）

## 一句话简介
主导设计并实现基于 LangGraph 的多智能体时序决策系统，围绕多阶段决策工作流、记忆闭环、规则校验与可观测性建设，提升复杂规则环境下 LLM Agent 的可执行性、稳定性与可复盘性。

## 简历版项目描述
- 主导设计基于 LangGraph 的多智能体时序决策架构，搭建红方快慢双系统：快层使用规则与启发式策略保障高时效响应，慢层通过 `analyst -> commander -> allocator -> operator -> critic` 五节点工作流完成态势分析、意图生成、资源分配、动作生成与结果校验，提升复杂时序环境中的决策稳定性与可解释性。
- 负责搭建 Agent 记忆闭环，设计短期记忆窗口聚合最近态势变化、威胁波动与关键事件，并实现长期经验检索机制，将历史 lessons 回流到当前决策；同时构建战后 reflection 流程，自动基于 battle log 生成 success/failure lessons 并结构化写回知识库。
- 负责落地 Agent 可靠性机制，接入结构化输出、分角色模型调用与 JSON 解析链路，叠加动作语义校验、critic 重试、freshness guard、task board 和重规划退避控制，降低模型输出在动态环境中的失效、过期与上下文漂移问题。
- 完成系统可观测性建设，沉淀决策日志、battle log、统一 trace 与 LLM 调用日志，仓库内已保留 219 份 battle logs、146 份运行日志、4088 份 LLM 输出文件，为策略复盘、Prompt 迭代和异常定位提供可追踪证据。

## 技术栈关键词
Python / LangGraph / LLM orchestration / Prompt engineering / Structured output / Memory / Reflection / Multi-agent workflow / Async scheduling / Observability

## 90 秒项目讲法
这个项目本质上不是一个简单的 prompt demo，而是一个面向复杂规则环境的多智能体决策系统。我主导设计了红蓝双方的决策框架，其中红方采用快慢双系统：快层用规则和启发式策略保证高时效响应，慢层用 LangGraph 串起 `analyst -> commander -> allocator -> operator -> critic` 五阶段工作流，让模型先做态势理解，再做意图生成、资源分配、动作生成和结果校验。为了让 Agent 不只是“会想”，还要“能稳定执行”，我又补了短期记忆、长期经验检索、战后反思写回、动作时效校验和语义过滤这些机制。这样系统就形成了“运行中决策 - 事后复盘 - 经验回流”的闭环，也更接近真实 Agent 工程，而不是一次性的模型调用。

## 核心难点
最大的难点不是“让模型给出答案”，而是“让模型在动态、强约束、时序变化快的环境里持续给出可执行、不过期、可复盘的答案”。

可展开回答：
- 复杂环境里，模型输出很容易出现格式正确但语义不可执行的问题。
- 环境持续变化后，上一时刻生成的动作可能已经失效，必须做时效校验和重规划。
- 只靠 Prompt 很难解决可靠性问题，所以我把规则层、校验层、记忆层和日志追踪层一起补齐，形成多层防护。

## 最能体现 Agent 工程能力的 Trade-off
我没有把所有决策都交给 LLM，而是采用“快层规则保障时效，慢层 LLM 负责高价值推理；确定性转换优先，模型生成作为回退”的设计。

这样取舍的原因：
- 纯 LLM 虽然灵活，但在高频时序环境里成本更高、延迟更大、稳定性更差。
- 纯规则虽然稳定，但难以处理复杂态势理解和资源分配。
- 快慢双系统和 deterministic-first 方案，本质上是在响应速度、成本和决策质量之间做工程化平衡，这比单纯堆 Prompt 更能体现 Agent 工程能力。

## 高频追问与答题框架
### 1. 你为什么选择 LangGraph？
- 回答框架：先说场景需要多阶段决策和状态传递，再说 LangGraph 适合显式编排节点、状态和重试路径，最后落到本项目里五节点工作流和 critic 回路是怎么实现的。

### 2. 这个项目最像 Agent 的地方是什么？
- 回答框架：不要只说用了 LLM，要强调它具备任务分阶段、状态持久化、记忆回流、校验反馈和复盘写回这些闭环能力，说明它是“可持续运行的系统”而不是一次性调用。

### 3. 你怎么保证模型输出稳定可执行？
- 回答框架：从四层讲，分别是结构化输出、语义校验、时效保护、失败重试；最后补一句，真正提高稳定性的不是单点技巧，而是多层机制叠加。

### 4. 短期记忆和长期记忆分别解决了什么问题？
- 回答框架：短期记忆解决当前窗口内的态势连续性问题，让模型知道最近发生了什么；长期记忆解决跨局经验复用问题，把历史复盘 lessons 检索回来影响当前决策。

### 5. 如果继续迭代，你下一步会优化什么？
- 回答框架：可以从评测体系、检索质量、决策回放工具三个方向讲；重点说你会先补离线评估和回放分析，因为 Agent 工程的核心不是“再加一个模型”，而是让系统可测、可调、可比较。

## 证据锚点
### 多阶段工作流与编排
- `graph.py:883-1077`：定义 `analyst`、`commander`、`allocator`、`operator`、`critic` 五节点，以及 critic 校验后的条件路由。
- `base_commander.py:54-170`：初始化 LangGraph System2、异步事件循环、STM/LTM/FreshnessGuard 和 trace 体系，体现慢层编排与工程化封装。

### 记忆闭环
- `memory_manager.py:30-116`：短期记忆窗口记录目标变化、威胁波动、关键事件，并生成窗口摘要与事件时间线。
- `ltm_retriever.py:31-127`：长期经验检索从结构化 lessons 中做 hybrid scoring，并将相关经验回流到当前上下文。
- `reflection_agent.py:35-203`：战后根据 battle log 自动生成 reflection，并以结构化 lessons 形式写回知识库。

### 可靠性与约束
- `llm_manager.py:28-50`：区分快慢模型、结构化输出、上下文缓存等运行参数。
- `llm_manager.py:435-522`：结构化生成支持 `json_schema`、`json_object`、`plain_text` 多模式降级与解析回退。
- `action_freshness_guard.py:18-62`：对动作时效、目标有效性与上下文漂移做过滤，并支持触发重规划。

### 可观测性
- `event_logger.py:11-145`：统一记录 battle event、decision 和 finalization 信息，形成 battle log。
- `test/battle_logs`：当前保留 219 份 battle logs。
- `test/logs`：当前保留 146 份运行日志。
- `llm_outputs`：当前保留 4088 份 LLM 输出文件。
